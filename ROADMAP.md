## 🚀 Implementation Methodology: TDD
All development must follow the **Test-Driven Development** cycle to ensure maximum code reliability and prevent regressions.

1.  **🔴 RED:** Write a failing unit/integration test that defines the expected behavior. No production code should exist yet for this specific feature.
2.  **🟢 GREEN:** Write the **minimum** amount of production code required to make the test pass.
3.  **🔵 REFACTOR:** Clean up the code, improve naming, and optimize structure **while ensuring all tests remain green**.

- [x] **10.5.B Unit-test parallelism — measured, intentionally not enabled** (2026-04-19). Chose Gradle `maxParallelForks` over JUnit 5 within-process threading (safer for the Robolectric + Hilt + Room stack — each JVM fork owns an isolated graph, no ThreadLocal collision risk). Measured on a 12-core Mac with `maxParallelForks = 6` (half of physical cores): **baseline 29.5 s → parallel 48.0 s, a 63 % regression**. Diagnosis: JVM fork startup (~3–5 s × 6) + Robolectric per-fork shadow-classloader init exceeds the parallelism gain at a 442-test / ~29 s suite size. Config change reverted; a sentinel comment left at [`app/build.gradle.kts`](app/build.gradle.kts) under `testOptions` explaining the measurement + revisit threshold (~2–3 min serial runtime). **Honest framing:** the ROADMAP proposed parallelism based on suite-count heuristic (442 tests); the real limiting factor is wall-clock cost, which is already low for this project. Not every ROADMAP-item is a code change — sometimes the right outcome is "measured, decided against, documented for future." 442 unit tests still green on the reverted (serial) config.
